# Rakuten Select Assist - 開発ルール

最終更新: 2026-03-31

---

## 1. このプロジェクトの目的

本プロジェクトは、楽天アフィリエイト利用者が抱える
**「商品選定に時間がかかりすぎる問題」**を解決するための
商品選定支援アプリを開発する。

初期フェーズでは **楽天市場向けWebアプリ** を第一対象とし、
Androidアプリ化や楽天トラベル対応は後続フェーズで再評価する。

---

## 2. 絶対ルール

1. **コード即書き禁止**  
   実装前に課題定義・有識者議論・方針決定を行うこと。

2. **MVP優先**  
   初期フェーズでは「商品選定支援」に集中し、投稿自動化・分析・高度なAI機能は後回しにする。

3. **スコープ厳守**  
   フェーズ外の要望は BACKLOG.md に記録し、今のタスクに混ぜない。

4. **ハルシネーション禁止**  
   ファイル名、関数名、API仕様、設定値は必ず実在確認してから扱う。

5. **APIキー保護**  
   APIキー・シークレット・トークンはクライアント側に直書きしない。公開リポジトリへ絶対に含めない。

6. **スクレイピング前提禁止**  
   楽天の公式APIで取得可能な情報は、必ず公式APIを優先する。

7. **技術的異議の明示**  
   問題のある方針には、理由を添えて反論する。忖度しない。

8. **品質妥協禁止**  
   動くだけの雑な実装を許可しない。保守性を重視する。

9. **Single Source of Truth**  
   開発判断は会話ではなく docs 配下のドキュメントを正とする。

---

## 3. フェーズ方針

### Phase 1
- 楽天市場の商品候補取得
- スコアリング
- 上位候補の提示
- 保存機能
- Webアプリとして公開可能な最小構成

### Phase 2
- AIによる紹介文補助
- トーン切り替え
- 選定理由の自然文生成

### Phase 3
- Threads連携
- 投稿補助
- 簡易分析

### Phase 4
- 楽天トラベル対応
- Androidアプリ検討

## 4. 有識者議論プロセス

### 4.1 基本原則
大きな実装や設計変更の前には、有識者視点で議論してから進める。

### 4.2 召喚する有識者
作業内容に応じて、最低3名以上の視点を使う。

| ペルソナ | 視点 | 主な場面 |
|----------|------|----------|
| アーキテクト | 全体設計・依存関係・拡張性 | 構成設計、技術選定 |
| セキュリティ専門家 | APIキー管理・脆弱性・認証 | 外部API、認証、保存 |
| UXデザイナー | 操作フロー・迷わない導線 | 画面設計、フォーム設計 |
| QAエンジニア | テスト容易性・回帰防止 | ロジック追加、仕様変更 |
| ビジネス視点 | MVP妥当性・費用対効果 | 優先順位、機能要否 |

### 4.3 議論の流れ
1. 課題定義
2. 有識者召喚
3. 論点整理
4. スコアリング
5. 8割以上なら着手
6. 結果を DECISIONS.md に記録

### 4.4 スコアリング基準

| # | 評価項目 | 配点 |
|---|----------|------|
| 1 | 技術的妥当性 | /20 |
| 2 | 保守性・可読性 | /15 |
| 3 | MVP適合性 | /15 |
| 4 | テスト容易性 | /10 |
| 5 | セキュリティ・運用安全性 | /15 |
| 6 | ユーザー価値 | /15 |
| 7 | 費用対効果 | /10 |
| | **合計** | **/100** |

**80点以上で着手許可**

## 6. ドキュメント運用ルール

### 必須更新対象
- DEVELOPMENT_PLAN.md  
  現在地、進捗、Next Steps を更新する

- DECISIONS.md  
  設計判断や技術選定理由を記録する

- BACKLOG.md  
  スコープ外だが重要な提案を記録する

### 記録しないもの
- 一時的な思いつき
- 実装に影響しない雑談
- 未確認の仮説

## 7. 技術選定ルール

### 第一候補
- フロントエンド: Next.js
- バックエンド: Next.js API Routes または Server Actions
- データ保存: Supabase または最小限のDB
- デプロイ: Vercel
- コード管理: GitHub

### 原則
- 新技術導入は「必要だから使う」。好みで増やさない。
- 初期フェーズでは依存を増やしすぎない。
- Androidネイティブ化は、MVP成立後に再評価する。

---

## 8. 楽天API・外部APIルール

1. 楽天関連データ取得は、公式APIを最優先する。
2. API仕様は必ず公式ドキュメントで確認する。
3. アフィリエイト導線やリンク生成は利用規約を確認の上で実装する。
4. 非公式スクレイピングに依存しない。
5. AI APIは「必須機能」ではなく「補助機能」として扱う。
6. AI API導入時は、コスト・失敗時のフォールバック・利用目的を明記する。

---

## 9. セキュリティルール

- `.env` をGit管理しない
- 公開クライアントに秘密情報を含めない
- サーバー側でAPI呼び出しを行う
- 入力値を検証する
- ログに機密情報を出さない
- URL生成時は不正な値混入に注意する

---

## 10. 実装前の影響範囲宣言

変更前に以下を明示すること。

1. 変更対象ファイル
2. 関連影響ファイル
3. 既存機能への影響
4. テスト対象
5. ロールバック方針

5ファイル以上に影響する変更は、要注意変更として扱う。

---

## 11. テストルール

- 新規ロジックには基本的にテストを追加する
- スコアリング関数は単体テスト対象にする
- API接続部分はモック可能な構造にする
- UIは重要導線を中心に確認する
- 「楽天API障害時」の代替挙動も考慮する
